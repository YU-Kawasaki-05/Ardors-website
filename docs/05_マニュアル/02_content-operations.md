# コンテンツ登録・更新手順

## 目的

公開前後に必要になるコンテンツ登録・更新作業をまとめます。

対象は以下です。

- 実績
- Note 記事
- プロフィール
- サービス
- SaaS 構想
- 法務ページ
- OGP 画像

## 1. 実績を登録・更新する

### 1.1 現在の推奨運用

本番永続ストアが未実装のため、Production に反映する実績は `data/works.json` を編集して PR 経由で反映します。

管理画面 `/admin/cases` はローカル・Preview での入力体験確認には使えます。ただし Vercel Hobby 本番での file 書き込みは永続運用の前提にしません。

### 1.2 実績1件に必要な情報

| 項目          | 必須 | 説明                                     |
| ------------- | ---- | ---------------------------------------- |
| `id`          | 必須 | 一意のID。既存と重複させない             |
| `slug`        | 必須 | URL 用。小文字英数字とハイフン           |
| `thumbnail`   | 任意 | OGP / 一覧用画像パス。未確定なら空でよい |
| `publishedAt` | 必須 | `YYYY-MM` 形式                           |
| `published`   | 必須 | 公開するなら `true`                      |
| `techStack`   | 必須 | 使用技術の配列                           |
| `nextWorks`   | 任意 | 詳細ページ下部の次導線                   |
| `locales.ja`  | 必須 | 日本語コンテンツ                         |
| `locales.en`  | 任意 | 英語コンテンツ。未完成なら省略           |

### 1.3 BR-10 の必須3要素

各実績には以下を必ず入れます。

| 項目       | 意味           |
| ---------- | -------------- |
| `problem`  | 課題           |
| `solution` | 実施した解決策 |
| `result`   | 成果・結果     |

この3要素が薄いと実績ページの説得力が落ちます。単なる作業一覧ではなく、事業・ユーザー・成果の流れで書きます。

### 1.4 JA/EN の扱い

- `locales.ja` は必須です。
- `locales.en` は任意です。
- 英語が未完成の実績は EN サイトに出しません。
- BR-21 のため、未翻訳の日本語を EN に仮置きしません。

### 1.5 実績更新後の確認

```bash
pnpm lint
pnpm typecheck
pnpm build
```

手動確認は以下です。

- `/works` に公開対象だけが出る。
- `/works/[slug]` が表示できる。
- 非公開の実績は公開側に出ない。
- EN 未入力の実績は EN サイトに出ない。
- 404 にすべき slug は 404 になる。

## 2. Note 記事を更新する

Note 記事はこのリポジトリで本文管理しません。

運用は以下です。

1. Note 側で記事を作成・公開する。
2. `NOTE_RSS_URL` が設定済みであることを確認する。
3. `/notes` で記事一覧に反映されることを確認する。
4. 反映されない場合は ISR の再取得タイミングを待つ。目安は最大1時間です。

記事カードには以下が表示されます。

- タイトル
- 公開日
- 抜粋
- Note 元記事への外部リンク

## 3. プロフィールを更新する

プロフィール情報は `data/profile.ts` を更新します。

確認観点は以下です。

- 日本語・英語の両方に不自然な未翻訳がない。
- 肩書き、自己紹介、強み、経歴が現在の実態と合っている。
- 実績やサービスとの表現が矛盾しない。

確認ページは `/profile` です。

## 4. サービス内容を更新する

サービス情報は `data/services.ts` を更新します。

確認観点は以下です。

- 誰向けのサービスかが明確である。
- 提供範囲と対象外が曖昧でない。
- CTA が問い合わせにつながっている。
- 実績ページと表現が矛盾しない。

確認ページは `/services` です。

## 5. SaaS 構想を更新する

SaaS 情報は `data/saas.ts` を更新します。

確認観点は以下です。

- まだ構想段階のものを、完成済みプロダクトのように書かない。
- 検証中・準備中・将来構想の区別を明確にする。
- 問い合わせや相談導線が自然につながっている。

確認ページは `/saas` です。

## 6. 法務ページを更新する

法務ページのデータは以下です。

| ページ             | データファイル            |
| ------------------ | ------------------------- |
| `/privacy`         | `data/legal/privacy.ts`   |
| `/terms`           | `data/legal/terms.ts`     |
| `/legal/tokushoho` | `data/legal/tokushoho.ts` |

法務文言は安易に変更しないでください。事業内容、販売方法、問い合わせ先、個人情報の扱いが変わったときに見直します。

## 7. OGP 画像を更新する

既定の OGP 画像は `public/og-default.png` です。

更新時の確認観点は以下です。

- 1200 x 630 px を基準にする。
- 小さく表示されても Ardors だと分かる。
- 文字を詰め込みすぎない。
- 背景と文字のコントラストを十分に取る。
- `pnpm build` 後、主要ページの OGP metadata が壊れていないことを確認する。

## 8. コンテンツ更新時の共通チェック

```bash
pnpm lint
pnpm typecheck
pnpm build
```

手動で確認するページは以下です。

- `/`
- `/services`
- `/works`
- `/works/[slug]`
- `/profile`
- `/saas`
- `/notes`
- `/contact`
- `/privacy`
- `/terms`
- `/legal/tokushoho`

確認観点は以下です。

- 各ページの h1 が1つだけである。
- JA/EN の切り替えで未翻訳の仮文字列が出ない。
- CTA が問い合わせに向いている。
- 外部リンクは別タブで開く。
- モバイル幅で横スクロールしない。
