# ユーザー像定義書

### 1. ユーザー種別一覧
| ID | ユーザー種別 | 説明 | 想定人数 | 利用頻度 |
|----|------------|------|---------|---------|
| U-01 | 見込み顧客 | Ardorsへ制作/技術相談を検討している訪問者 | 多 | 案件検討時 |
| U-02 | 知人・取引先 | 既存のつながりで活動確認や紹介判断をする訪問者 | 中 | 必要時 |
| U-03 | 協業希望者 | 一緒に仕事をしたい、共同提案したい訪問者 | 中 | 協業検討時 |
| U-04 | 志望先企業担当者 | あなたの能力・実績・職務適性を確認する担当者 | 低〜中 | 選考時 |
| U-05 | サイト運営者（あなた） | コンテンツ更新や問い合わせ対応を行う管理側ユーザー | 1 | 週次〜随時 |

### 2. ペルソナ詳細

#### U-01: 見込み顧客
- **役割**: 発注・相談の意思決定者、または実務担当者
- **主な利用目的**: Ardorsの提供価値・実績・相談可否を短時間で判断
- **利用シーン**: 紹介後、検索経由、SNS経由で初訪問
- **利用デバイス**: PC/スマホ
- **ITリテラシー**: 中〜高
- **主要タスク**:
  1. サービス内容の把握
  2. 実績確認
  3. 問い合わせ送信
- **ペインポイント**: 何が依頼できるか不明だと離脱しやすい
- **成功体験**: 数分で理解し、安心して相談を送れる

#### U-02: 知人・取引先
- **役割**: 信頼できる紹介元・既存関係者
- **主な利用目的**: 現在の活動や実績を確認し、他者へ紹介できる状態にする
- **利用シーン**: 近況確認、紹介前の確認
- **利用デバイス**: スマホ中心
- **ITリテラシー**: 中
- **主要タスク**:
  1. プロフィール確認
  2. 実績確認
  3. 共有リンク送付
- **ペインポイント**: 情報が散在すると紹介しづらい
- **成功体験**: 1URLで十分説明できる

#### U-03: 協業希望者
- **役割**: パートナー候補（開発会社、個人、専門職）
- **主な利用目的**: 技術領域・実績・協業余地の把握
- **利用シーン**: 協業相談前
- **利用デバイス**: PC中心
- **ITリテラシー**: 高
- **主要タスク**:
  1. 技術強み確認
  2. 実績と守備範囲確認
  3. 相談導線利用
- **ペインポイント**: 得意領域が曖昧だと相談に進まない
- **成功体験**: 役割分担が想像でき、具体相談に進める

#### U-04: 志望先企業担当者
- **役割**: 採用・配属判断に関わる担当者
- **主な利用目的**: あなたの経験、実装力、課題解決力の確認
- **利用シーン**: 選考中、面談前後
- **利用デバイス**: PC
- **ITリテラシー**: 中〜高
- **主要タスク**:
  1. プロフィール確認
  2. 実績確認
  3. GitHub等の外部実績確認
- **ペインポイント**: 成果や関与範囲が不透明だと評価しづらい
- **成功体験**: 強みと実績が構造的に把握できる

#### U-05: サイト運営者（あなた）
- **役割**: コンテンツ管理者
- **主な利用目的**: 実績情報やプロフィールを継続更新し、問い合わせ対応を行う
- **利用シーン**: 週次更新、案件完了後、必要時
- **利用デバイス**: PC
- **ITリテラシー**: 高
- **主要タスク**:
  1. 事例情報の登録/編集（初期P1）
  2. リンク/導線メンテナンス
  3. 問い合わせ確認
- **ペインポイント**: 更新運用が重いとサイトが陳腐化する
- **成功体験**: 少ない工数で最新状態を維持できる

### 4. 認証方針（現時点）
- 公開サイト閲覧側にログインは不要（確定）。
- 事例紹介CMSを実装する場合、管理操作は U-05 のみ許可する。
- 管理画面は公開導線と分離し、`/admin/*` 配下に隔離する。

### 5. セキュリティ観点（要求）
- セキュリティ要件は高水準とする（ユーザー要望）。
- 公開入力（問い合わせフォーム）は不正入力対策を前提に設計する。
- 具体基準は確定済み（入力検証、レート制限、セキュアヘッダー、脆弱性閾値、監視要件）。
