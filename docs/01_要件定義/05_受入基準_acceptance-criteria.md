# 受け入れ基準書

## 1. 概要
本書は Ardors Webサイトの機能受け入れ基準を定義する。  
優先度 `P0` は詳細シナリオで定義し、`P1` は主要シナリオのみ定義する。

## 2. 前提
- 判定対象環境は本番相当（HTTPS有効、計測タグ有効、実データ同等）とする。
- ブラウザは `Chrome / Safari / Firefox / Edge` の最新版を対象とする。
- モバイルは `iOS Safari` と `Android Chrome` の最新版を対象とする。

## 3. 受け入れ基準（P0）

### FR-01: トップページ
**AC-01-01: 正常系 - ヒーロー要件表示**
```gherkin
Given ユーザーが "/" にアクセスする
When トップページが表示される
Then ファーストビューに「提供対象」「提供価値」「問い合わせ導線」が表示される
And 主要CTAがファーストビューとページ下部の2箇所以上に表示される
```

**AC-01-02: 正常系 - 入口分岐導線**
```gherkin
Given ユーザーがトップページを閲覧している
When ファーストビュー直下を確認する
Then 「相談したい」「実績を見たい」「人物を知りたい」の3導線が表示される
And 各導線は対応画面へ遷移する
```

### FR-20: 問い合わせフォーム
**AC-20-01: 正常系 - 送信成功**
```gherkin
Given ユーザーが "/contact" を表示している
And 名前、メールアドレス、相談種別、本文を有効値で入力している
When 「送信」ボタンをクリックする
Then 問い合わせデータが保存される
And "/contact/complete" に遷移する
```

**AC-20-02: 異常系 - 必須未入力**
```gherkin
Given ユーザーが "/contact" を表示している
When 必須項目を空で送信する
Then 送信は実行されない
And 未入力項目ごとにエラーメッセージが表示される
```

**AC-20-03: バリデーション - 本文文字数**
```gherkin
Given ユーザーが "/contact" を表示している
When 本文の文字数が制約外で送信する
Then 文字数エラーが表示される
And 送信は実行されない

Examples:
  | 本文文字数 | 判定 |
  | 0          | NG   |
  | 1          | OK   |
  | 3000       | OK   |
  | 3001       | NG   |
```

### FR-22 / FR-70 / FR-71: フォーム保護・入力検証・セキュアヘッダー
**AC-22-01: セキュリティ - サーバー側入力検証**
```gherkin
Given 改ざんされたHTTPリクエストが送信される
When サーバーが問い合わせリクエストを受信する
Then サーバー側で形式/長さ/必須が検証される
And 不正データは保存されない
And HTTPステータス400系を返す
```

**AC-22-02: セキュリティ - レート制限**
```gherkin
Given 同一IPから短時間にフォーム送信が行われる
When 1分間で6件目の送信を試行する
Then レート制限エラーが返る
And 6件目以降の保存は行われない
```

**AC-22-03: セキュリティ - ヘッダー適用**
```gherkin
Given ユーザーが公開ページへアクセスする
When HTTPレスポンスヘッダーを確認する
Then CSP, HSTS, X-Frame-Options, X-Content-Type-Options が設定されている
```

**AC-22-04: セキュリティ - 内部情報非公開**
```gherkin
Given サーバー内部で例外が発生する
When ユーザーにエラーレスポンスを返す
Then レスポンス本文にスタックトレースや秘密値を含めない
```

## 5. 非機能要件の受け入れ基準

### AC-NFR-01: セキュリティ
- 問い合わせフォームに対し、SQLi/XSS/CSRFの基本攻撃パターン送信時に入力保存されないこと。
- HTTPSを強制し、HTTPアクセス時はHTTPSへリダイレクトすること。
- 依存パッケージの既知脆弱性（CVSS 7.0以上）は本番反映前に0件であること。

### AC-NFR-02: 可用性・信頼性
- 月間稼働率 99.5%以上。
- 問い合わせ送信APIの5xx率が1%未満（週次集計）であること。

### AC-NFR-03: パフォーマンス
- 主要公開ページの初回表示時間（TTFB+描画）が3秒以内（モバイル4G相当）。
- 画像最適化未実施によるLCP劣化が発生しないこと。

### AC-NFR-04: UX/アクセシビリティ
- 主要導線が2タップ以内で到達できること（モバイル）。
- キーボードのみで問い合わせ完了まで到達できること。
- 日本語/英語の主要導線ラベルが意味的に一致すること。

## 6. 判定方法
- 受け入れ判定は `Pass / Fail / 保留` の3状態で記録する。
- `P0` は全項目 `Pass` をリリース条件とする。
- `P1` は初回リリース時に `保留` を許容するが、次フェーズ計画へ引き継ぐ。
